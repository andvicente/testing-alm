Testing ALM - Exemplos de Testes Funcionais com ferramentas Open Source

Download this project as a .zip file Download this project as a tar.gz file

Introdução

Este projeto foi criado para uma palestra no TDC 2012 em SP no dia 05/07/2012 na trilha de ALM com o título: Testes Funcionais em um ALM Open Source

O intuito do exemplo e apresentação é mostrar que, dentro de um projeto, todos devem dar foco não somente aos testes unitários e integração de componentes, mas também aos testes funcionais. Hoje muitos desenvolvedores usam, de forma correta, uma gama de ferramentas de teste e relacionadas para garantir a qualidade do código, mas isso não basta para entregar um produto de qualidade ao cliente.

Proposta

Apresentar a integração das ferramentas de Testes Funcionais e ferramentas relacionadas a etapa de teste. O projeto parte do principio que o time de teste utilize ferramentas para as seguintes frentes:

  • Gestão de Teste
  • Gestão de Defeitos
  • Automação Funcional / Aceitação

Funcionamento

Existem quatro pacotes relacionados as ferramentas de teste:

  • bugzilla
  • mantis
  • testlink
  • redmine

Cada pacote contem duas classes onde é o nome da ferramenta:

  • Instance: classe responsável pela comunicação inicial com a ferramenta (Singleton)
  • Utils: classe de reescrita dos métodos da API de cada ferramenta

O pacote selenium contém os testes criados com Selenium/Webdriver utilizando um design pattern chamado PageObjects

O pacote utils possui classes de apoio as ferramentas e a execução de teste

O pacote testes possui o teste que é apresentado como exemplo

Configuração inicial

Antes de executar os testes deste projeto é necessário alterar o arquivi config.properties com as URL's e acessos necessários para cada ferramenta.

Após isso é necessário configurar o Testlink e alguma ferramenta de Gestão de Defeitos para executar um dos exemplos. As configurações das ferramentas encontram-se no final desta página.

Configuração das Ferramentas

Testlink

Foi utilizada a versão 1.9.3 para a integração do Testlink. Utilizei a API Java desenvolvida por Bruno Kinoshita achamada testlink-java-api

Para que seja possível executar o exemplo é necessário:

  • Efetuar login no Testlink
  • Ir ao menu My Settings e clicar no botão Generate API Access Key. Guarde esta chave
  • Adicionar as seguintes linhas no arquivo custom_config.php ** $tlCfg->api->enabled = TRUE; ** $tlCfg->exec_cfg->enable_test_automation = ENABLED;
  • Alterar as seguintes propriedades o arquivo config.properties da aplicação de exemplo ** testlink.url: substitua pela URL do Testlink sem remover o restante do caminho ** testlink.devkey: cole a key gerada ao item 2

Mantis

Foi utilizado a versão 1.2.8 para a integração do Mantis. Utilizei a API Java chamada MantisConnect, mantida por um dos desenvolvedores do Mantis

Para que seja possível executar o exemplo é necessário:

  • Alterar as seguintes propriedades no arquivo config.properties da aplicação do exemplo ** mantis.url: substitua pela URL do Mantis sem remover o restante do caminho ** mantis.user: usuario registrado no Mantis ** mantis.password: senha do usuário registrado no Mantis

Bugzilla

Foi utilizada a versão 4.2.1 para a integração do Bugzilla Nenhuma API especifica do Bugzilla foi usada. Utilizei diretamenta a classe XmlRpcClient da Apache para submeter as informações ao Bugzilla.

Para que seja possível executar o exemplo é necessário:

  • Alterar as seguintes propriedades no arquivo config.properties da aplicação do exemplo ** bugzilla.url: substitua pela URL do Bugzilla sem remover o restante do caminho ** bugzilla.user: usuario registrado no Bugzilla ** bugzilla.password: senha do usuário registrado no Bugzilla

Redmine

Foi utilizada a versão 2.0.3 para a integração do Redmine Utilizei a API redmine-java-api mantida pelo grupo de desenvolvimento do Redmine para submeter as informações para o Redmine.

Como o Redmine não é nem um ferramenta de gestão de teste e nem uma ferramenta de gestão de defeitos, aqui estamos utilizando o módulo de issues, que se assemelha mais ao modelo de bugs.

Para que seja possível executar o exemplo é necessário

  • Efetuar login no Redmine
  • Clicar no link My account
  • Clicar no link Show do item API access key
  • Alterar as seguintes propriedades no arquivo config.properties da aplicação do exemplo ** redmine.url: substitua pela URL do Redmine sem remover o restante do caminho ** redmine.key: cole a chave do copiada do item API access key

Execução do Exemplo

O exemplo está localizado no pacote com.eliasnogueira.testing-alm.testes e a classe de testes que deve ser executada é ExecutaTestesPorFerramenta

Esta classe utiliza o JUnit para o suporte a execução dos testes e tem três métodos de teste:

  • erroBugzilla: executa o teste e reporta um bug no Bugzilla
  • erroRedmine: executa o teste e reporta uma issue no Redmine
  • erroMantis: execute o teste e reporta um erro no Mantis

Todos os métodos de teste também reportam resultado para o Testlink.

O seguinte trecho em cada método irá gerar uma exception para simular um erro e chamar a integração de cada ferramenta:

assertTrue(login.estaNaPaginaDeLogin("PROVOCA ERRO"));

O resultado esperado para este assertion seria ALM TDC
